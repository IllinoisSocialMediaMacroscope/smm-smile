path required path denotes the url of your analysis page. For example, {"path": "/NLP-sentiment", ...} defines that the sentiment page will show up if you route to https://socialmediamacroscope.org/.../.../NLP-sentiment
title required title will be displayed on the top of an analysis page
imgURL required imgURL will be the icon for that analysis tab in the home page.
wiki not required this will be the url to redirect to after clicking "read more..." button located within the analysis tab in the home page
introduction required this is the short paragraph describing the analysis in the home page. Long sentences can be break into segments and SMILE will be able to concatenate them by space.
result_path required It defines where your output is stored on the remote storage S3 bucket. Note you only need to provide a relative path, i.e. SMILE has already taken into account that which S3 bucket and under which user/session the output will store. For example, in run time {"result_path": "/NLP/sentiment/", ...} means my (username: cwang138) sentiment analysis results will be stored under the path /SMILE/cwang138/NLP/sentiment/
args required It is the ID of the input parameter, if your algorithm has no input parameters, you can simply leave the value of args as an empty array. But it is very important that each value of the args must match the id of html element. . For example: {"args": ["algorithm"], ...} means there will be one user input parameter and its id will be "algorithm". This matches the part in the same config file which we will talk about later. { "get": "containers": [{ "input": { "type": "select", "name": "algorithm", "id": "algorithm", ... }, }] ... }
custom_script not required The value of this entry should be the path of a custom javascript file, for example: {"custom_script": "bootstrap/js/customized/analyses/view_classification.js", ...}. Within this .js file you can add your own custom actions to overwrite the current actions, or to add more effects to the frontend.
get not required
containers
input Input the actual input element, which requires a type, name, id and based on what type it is there will be more fields
buttonGroups represents the buttons you see by the end of the form. If your analysis just need a simple Submit/Clear feature that POST all the form input to the backend, you should use the template.
citation defines the citation block content you will see on the page. You need to decide which event triggers which citation display. trigger_id is the element id that you want to listen to, for example, setting the "trigger_id" to be "algorithm" means whenever I change algorithm, the citation blocks will toggle and change. text is the full text content of a citation, condition is the condition to evaluate and decides which citation to show.
